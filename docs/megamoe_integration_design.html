FlyDSL MegaMoE 移植到 rtp-llm 集成实现（as-built）

目标模型：Qwen3.5-397B-A17B  ·  硬件：AMD Instinct MI308X ×8 (gfx942 / CDNA3)  ·  EP8 / TP8  ·  初稿：2026-07-15  ·  按实际实现更新：2026-07-22
文档目的：把 FlyDSL 中已验证的「2-stage 融合 MegaMoE」方案（stage1 = dispatch+GEMM1 融合，stage2 = GEMM2+combine 融合，e2e 全档 1.29×~1.73× 反超 aiter CK 基线）移植进 rtp-llm 推理引擎，通过环境变量开关控制启用，补充单元测试，并在 Qwen3.5-397B-A17B 上完成端到端验证。本文最初为设计评审稿，现已按实际落地代码回填修订：架构骨架（方案 B）与设计一致，但组件命名、开关形态、权重注入方式、约束边界（topk）等细节以本文当前描述为准。文中标注 与设计不同 处即实现阶段的实际偏离。

1背景与目标

已有成果（FlyDSL 侧）

在 FlyDSL/kernels/ 中已实现并验证 gfx942 plain-FP8 的两段式融合 MoE：

  • Stage1：mori dispatch + GEMM1(gate_up·silu·mul) 融合进单 kernel
  • Stage2：GEMM2(down) + P2P combine 融合进单 kernel
  • kernel 内直接 emit sorted 表（atom_sorted），e2e forward 零 host sync
  • 8 卡多 batch 精度全 PASS，支持 CUDAGraph

本次目标（rtp-llm 侧）

  • 把该方案作为一个新的 MoE 后端接入 rtp-llm 的 Strategy/Router/Executor 框架
  • 新增环境变量 USE_MEGAMOE 控制启用，与现有 Mori / DeepEP 互斥
  • 补充与现有 moriep_intranode_router_test.py 同风格的多卡单测
  • 集成进 GenericMoeLayer 端到端路径
  • 参考 start_mori.sh 产出 start_megamoe.sh，在 Qwen3.5-397B-A17B 上验证

2FlyDSL MegaMoE 方案架构

核心是一个 per-rank 的端到端模块 FusedMoEZeroCopyFp8（kernels/moe_fused_chained_fp8.py），把整条 MoE 前向压缩成 5 个 GPU 步骤，全程走 mori 对称显存做跨卡通信。

依赖：flydsl（MLIR DSL 编译器，最大外部依赖）mori.shmem（rtp-llm 已通过 MoriEP 引入）aiter（ROCm，已可用，用于 per-token quant）。约束：world_size ≤ 8（mori intra-node）、experts % world_size == 0、与设计不同 topk ≤ 64（warp-ballot 车道预算）——设计原写 topk ≤ world_size，实测发现该假设是遗留错误：combine 缓冲 shmem_comb_inp_tok 早已按 max(npes·mt, mt·topk+1) 分配，topk > npes 天然可放。正是放开此约束才跑通 Qwen3.5-397B-A17B（topk=10 > world_size=8）。另需 max_tok_per_rank 为 2 的幂。

3rtp-llm MoE 架构现状

rtp-llm 用 Strategy + Factory + Registry 模式组织 MoE，核心把 Router（跨卡 dispatch/combine）与 Executor（专家计算）解耦：

现有 ROCm 后端选择逻辑

RocmEpNormalStrategy 是一个策略根据 config 返回不同 router：

  • use_mori_ep=1 → MoriEpIntranodeRouter + aiter executor
  • use_deepep_moe=1 → DeepepNormalRouter + FusedMoeExecutor
  • 两者同时开 → 抛 ValueError（互斥）

优先级与注册

优先级 = router_type.value*10 + executor_type.value，自动计算。

设备相关策略在 fused_moe/__init__.py 内按 get_device_type() 注册。ROCm 分支已注册 5 个策略。新增后端在此 registry.register(...) 即可。

4核心难点：融合边界错配

关键矛盾：rtp-llm 把 MoE 切成 prepare(dispatch) → execute(GEMM) → finalize(combine) 三段；而 FlyDSL MegaMoE 把 dispatch+GEMM1 融进 stage1、GEMM2+combine 融进 stage2 —— 融合恰好跨越了 rtp-llm 的相位边界。无法按「router 只做 dispatch / executor 只做 GEMM / router 只做 combine」的方式拆开塞进去。

三种候选方案对比

方案做法评价
A. 纯 Router 独占prepare() 内部做完整条流水线，execute/finalize 空转违背框架语义，executor 形同虚设，量化配置解析链路割裂
B. Executor 独占 + 透传 Router ✅Router 只算路由(gate→topk)并透传张量；Executor 持有 FusedMoEZeroCopyFp8 跑完整流水线；finalize 透传结果最贴合：融合边界整体落在 executor 内，对框架侵入最小，量化/权重走 executor 既有通道
C. 双 kernel 拆分对齐把 stage1/stage2 拆开分别塞进 prepare/execute/finalize需要拆散已验证的融合 kernel，等于重做，丧失融合收益
选定方案 B：新增「透传式 MegaMoE Router」+「独占式 MegaMoE Executor」。Router 的 prepare 不做实际跨卡 dispatch（原样返回 hidden_states + 全局 topk_ids/weights），Executor 的 execute 调用 FusedMoEZeroCopyFp8.forward() 完成 dispatch+GEMM1+requant+GEMM2+combine，Router 的 finalize 直接透传 executor 已 combine 的输出。

6数据流与张量规格

Qwen3.5-397B-A17B 在 EP8 下的形状（需从模型 config 读取并校验）：

符号含义Qwen3.5-397B-A17B (EP8) 实测取值
model_dimhidden size4096
inter_dim专家 FFN 中间维1024（moe_intermediate_size）
experts全局专家数512（512 % 8 == 0 ✓）
epr每卡专家数 = experts/864
topk每 token 激活专家10（> world_size=8，靠放开约束支持）
max_tok_per_rank每卡 token 容量512（MEGAMOE_MAX_TOK 默认，向上取 2 的幂）

与设计不同：设计原假设 topk ≤ 8，实测 397B 的 topk=10 超过 world_size；由于约束放开到 topk ≤ 64（见 §2），无需回退 Mori/DeepEP，直接由 MegaMoE 服务。数值从 ~/.cache/modelscope/.../Qwen3.5-397B-A17B/config.json 读取，仍校验 experts % 8 == 0。

executor.execute 内张量映射

ExpertForwardPayload:
  expert_x            [cur_tok, model_dim]  bf16   ← 未 dispatch 的原始激活
  expert_topk_ids     [cur_tok, topk]       i32    ← 全局 expert id
  expert_topk_weights [cur_tok, topk]       f32
        │
        ▼   wrapper.forward(x_bf16=expert_x, wts=weights, topk_ids=ids, w1, scale_w1)
        │   (内部 quant→stage1→requant→stage2→combine)
        ▼
CombineForwardPayload:
  fused_expert_output [cur_tok, model_dim]  bf16   ← combine 已完成的最终输出
        │
        ▼   router.finalize() 透传
   最终 MoE 输出 → 回到 GenericMoeLayer
路由权重语义对齐：FlyDSL 的 combine 在 kernel 内做加权归约（doweight_fused），而 rtp-llm 的 finalize 通常也做加权。需确保权重只乘一次——设计上让 MegaMoE 全程持有 topk_weights 并在 combine 内应用，finalize 不再乘权重（apply_router_weight_on_input 语义需核对）。

7权重准备（preshuffle / 量化）

FlyDSL kernel 要求 W1/W2 为 fp8_e4m3fnuz + 16×16 preshuffle 布局，且带 per-N f32 scale。rtp-llm 的权重加载在模型 init 阶段完成，需要一个转换步骤：

权重原始（HF ckpt）MegaMoE 需要转换
W1 (gate_up)bf16 [E, 2·inter, model]fp8 [epr, 2·inter, model] preshuffled切本卡 epr + per-N quant + shuffle_weight(16×16)
W2 (down)bf16 [E, model, inter]fp8 [epr, model, inter] preshuffled同上
scale_w1/w2—f32 per-N量化时产出
设计决策 → 实际实现：与设计不同。设计想给 FusedMoEZeroCopyFp8 增一个「注入外部 W2」的构造入口；实际没有 set_w2/load_w2 API。op 在 _alloc_buffers() 里仍分配一块随机占位 W2（bf16→fp8 + scale=ones），executor 通过直接 copy_ 覆写 op 的 _w2_storage/_w2_scale 属性缓冲来注入真实权重。又因 op 是全局共享单例，executor 在每次 execute() 都把本层的 W2 重新灌回缓冲（否则"最后构造的层会赢"）。W1（gate_up）走 pertoken_quant + shuffle_weight 后 per-call 传入。量化/shuffle 辅助函数从 flydsl.tests.utils import。
依赖上移（未完成的技术债）：与设计不同。设计要求把 Stage2 wrapper FlyDSLMoeGemm2CombineOpGfx942（及 tests/utils 的 shuffle_weight/pertoken_quant）上移到 kernels/。实际仍留在 tests/kernels/test_profiler_moe_gemm2_combine.py，生产路径 kernels/moe_fused_chained_fp8.py 用 sys.path.insert 反向 import tests/——即生产代码依赖测试代码，属遗留技术债。
